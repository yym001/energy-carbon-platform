## 一、总则

### 1.1 适用范围

本手册适用于 AI 生成所有后台管理系统、业务功能页面、数据看板、统计分析类前端原型，核心覆盖**列表 CRUD、图表交互、表单管理、详情展示、数据看板**五大核心场景。

### 1.2 核心生成原则

1. **可运行原则**：所有原型必须为**单文件 HTML 格式**，CSS+JS 全内置，仅可引入开源稳定 CDN 资源（Chart.js、ECharts、Lucide 图标库），无本地依赖、无后端要求，浏览器直接打开即可完整运行。

2. **全交互原则**：所有标注的功能必须实现真实交互，点击、输入、切换、增删改查等操作必须有对应效果，禁止输出仅外观无操作能力的静态原型。

3. **数据闭环原则**：所有字段必须 100% 对应（表单字段↔列表字段↔详情字段），操作后数据实时更新，使用`localStorage`存储模拟数据，刷新页面数据不丢失。

4. **操作安全原则**：所有删除、修改、提交等不可逆操作，必须有二次确认机制；所有操作必须有明确的成功 / 失败 / 警告反馈。

5. **规范兼容原则**：所有组件样式、命名规范、交互逻辑必须完全兼容《UI 组件库 AI 维护规范手册》，禁止出现样式冲突、命名混乱。

6. **抽屉统一原则（最高优先级）**：

   - ✅ 必须用右侧抽屉的场景：新增表单、编辑表单、详情查看、数据明细、分步表单、高级筛选、批量操作设置、图表配置等**所有带业务内容、表单、长文本、多字段的弹出场景**
   - ⚠️ 仅可使用居中小弹窗的场景：删除二次确认、提交二次确认、操作成功 / 失败极简提示、风险警告确认**无输入内容的纯确认场景**
   - ❌ 绝对禁止：业务内容使用居中弹窗、抽屉与弹窗混用、无规范的弹出形式

## 二、右侧抽屉组件专属强制规范

本章节为所有抽屉组件的唯一执行标准，AI 生成时必须 100% 遵守，确保全系统抽屉交互、样式、结构完全统一，与 UI 组件库规范完全兼容。

### 2.1 抽屉固定结构（三区域必填）

所有右侧抽屉必须包含**头部固定区、内容滚动区、底部操作固定区**三大核心区域，结构不可缺失，布局不可错乱。

```
<!-- 抽屉标准结构模板，类名必须遵循drawer-前缀规范 -->
<div class="drawer-mask" id="drawerMask"></div>
<div class="drawer-container" id="targetDrawer">
    <!-- 头部固定区：必须固定在顶部，不随内容滚动 -->
    <div class="drawer-header">
        <div class="drawer-title">这里填写抽屉标题（如：新增设备、编辑账号、详情查看）</div>
        <button class="drawer-close-btn" id="drawerCloseBtn">
            <i data-lucide="x"></i>
        </button>
    </div>
    <!-- 内容滚动区：核心业务内容，超出高度自动滚动，不影响头部和底部 -->
    <div class="drawer-body">
        <!-- 表单、详情、业务内容全部写在此区域 -->
    </div>
    <!-- 底部操作固定区：必须固定在底部，不随内容滚动 -->
    <div class="drawer-footer">
        <button class="btn-secondary drawer-cancel-btn">取消</button>
        <button class="btn-primary drawer-confirm-btn">确认提交</button>
    </div>
</div>
```

### 2.2 标准尺寸规范

抽屉宽度固定为 3 种标准尺寸，根据业务内容复杂度选择，全系统统一，禁止随意自定义宽度。

| 尺寸等级 | 固定宽度 |                           适用场景                           |
| :------: | :------: | :----------------------------------------------------------: |
|  小尺寸  |  360px   |         短表单、简单详情、状态修改、少量字段的配置项         |
|  中尺寸  |  560px   | 常规新增 / 编辑表单、详情查看、10-20 个字段的业务表单（默认首选尺寸） |
|  大尺寸  |  720px   |   复杂分步表单、多模块详情、带关联数据的业务内容、宽表表单   |

- 强制要求：抽屉高度必须与浏览器视窗 100% 等高，禁止非通高抽屉；宽度固定，不可拖拽缩放。
- 叠加规则：禁止超过 2 层抽屉叠加，若需二级抽屉，宽度必须小于第一层（如第一层 560px，第二层 360px）。

### 2.3 样式与命名规范

1. **类名规范**：所有抽屉相关样式类名必须以`drawer-`为前缀，与 UI 组件库命名规范统一，禁止无意义类名。

2. **样式强制要求**：

3. **主题兼容**：所有抽屉样式必须复用组件库 A 区定义的全局 CSS 变量（主色、圆角、边框色等），禁止使用冲突配色。

### 2.4 交互强制规范

1. **打开 / 关闭动画**：抽屉打开必须有从右向左滑入的平滑动画（0.3s ease），关闭时有向右滑出的动画，禁止无动画的生硬切换。

2. **关闭方式（必须全部支持，形成交互闭环）**：

   - 点击右上角关闭按钮可关闭
   - 点击底部「取消」按钮可关闭
   - 点击遮罩层可关闭（表单类抽屉关闭前需校验是否有未保存内容，弹出二次确认）
   - 按下键盘 ESC 键可关闭

   

3. **数据回填规范**：编辑 / 详情类抽屉打开时，必须 100% 回填对应数据，字段与列表数据完全对应，无数据遗漏、错配。

4. **未保存提醒**：表单类抽屉，用户输入内容后未提交就关闭时，必须弹出二次确认弹窗，提示「您有未保存的内容，确认关闭吗？」，防止误操作丢失数据。

5. **提交后处理**：表单提交成功后，必须自动关闭抽屉，同时刷新列表数据，弹出「操作成功」反馈提示；提交失败时，保留抽屉打开状态，展示错误原因，支持重新提交。

6. **滚动规范**：抽屉内容超出视窗高度时，仅`drawer-body`内容区可滚动，头部和底部必须固定不动，禁止整体滚动。

7. **层级规范**：抽屉遮罩层级必须高于页面所有内容，抽屉容器层级高于遮罩，禁止出现被页面元素遮挡的情况。

## 三、核心场景功能详细要求（全场景抽屉适配）

### 3.1 列表类页面（增删改查全功能）

#### 3.1.1 核心要求

实现完整的「新增、删除、修改、查询、详情查看」全流程闭环，字段全对应，操作全确认，数据实时联动，浏览器打开即可直接操作全流程功能；**所有新增、编辑、详情、高级筛选功能必须使用右侧抽屉实现，仅删除操作使用居中二次确认弹窗**。

#### 3.1.2 必选功能

1. **基础列表渲染**
   - 表格结构完整，包含表头、数据行、分页控件，表头字段与业务数据字段 100% 对应，无字段缺失、错乱。
   - 内置真实模拟数据，不少于 10 条有效业务数据，数据符合业务场景逻辑，无空数据、无效值。
   - 基础样式：行 hover 高亮、斑马纹、选中行高亮、超长文本省略号 + tooltip、数据状态标签（正常 / 禁用 / 待审核 / 异常等）。
2. **查询功能**
   - 支持模糊搜索（按核心字段全局搜索）+ 基础条件筛选，高级筛选功能必须使用右侧抽屉实现，抽屉内包含完整的多条件组合筛选表单。
   - 必须包含「查询」「重置」两个核心按钮，点击查询后列表实时刷新匹配结果，点击重置清空所有筛选条件，恢复全量数据展示。
   - 支持多条件组合筛选，筛选结果准确无误，无数据遗漏、错配。
   - 已选筛选条件必须以标签形式可视化展示，支持单个条件快速清除。
3. **新增功能（必须使用右侧抽屉）**
   - 列表顶部必须有「新增」主按钮，点击打开**中尺寸右侧抽屉**，抽屉内包含完整的新增表单，表单字段与列表字段 100% 对应，必填项标 * 标识。
   - 表单必须包含完整校验逻辑：必填项非空校验、格式校验（手机号 / 邮箱 / 身份证 / 数字范围等）、实时失焦校验、错误提示清晰可见。
   - 校验不通过时，抽屉底部「确认提交」按钮禁用；校验通过后，点击提交必须弹出二次确认弹窗，明确提示「是否确认提交新增数据？」。
   - 确认提交后，数据实时新增至列表顶部，列表自动刷新，抽屉自动关闭，同时弹出「新增成功」的反馈提示。
   - 抽屉「取消」按钮、关闭按钮、点击遮罩关闭时，若表单有输入内容，必须弹出未保存二次确认，无内容则直接关闭。
4. **编辑功能（必须使用右侧抽屉）**
   - 列表每行操作栏必须有「编辑」按钮，点击打开**中尺寸右侧抽屉**，抽屉表单自动回填当前行的全量数据，与列表字段完全对应。
   - 编辑表单复用新增表单的校验规则，支持字段修改，不可编辑字段需设置为只读 / 禁用状态。
   - 点击提交必须弹出二次确认弹窗，明确提示「是否确认保存修改内容？」，确认后列表对应行数据实时更新，抽屉自动关闭，弹出「修改成功」反馈提示。
   - 取消编辑时，若表单有修改内容，必须弹出未保存二次确认，无修改则直接关闭。
5. **删除功能（仅此处可使用居中小弹窗）**
   - 列表每行操作栏必须有「删除」按钮，同时支持顶部「批量删除」按钮（批量删除按钮未勾选数据时默认禁用）。
   - 点击删除 / 批量删除，必须弹出**居中二次确认小弹窗**，明确提示「删除后数据不可恢复，是否确认删除选中的 X 条数据？」，禁止无确认直接删除，禁止使用抽屉实现删除确认。
   - 确认删除后，对应数据从列表中实时移除，列表自动刷新，弹出「删除成功」反馈提示；取消删除则无任何数据变更。
   - 禁用状态 / 不可删除的数据，删除按钮必须设置为禁用状态，并添加 tooltip 提示原因。
6. **详情查看功能（必须使用右侧抽屉）**
   - 列表每行必须支持「点击行」或「点击详情按钮」打开**对应尺寸的右侧抽屉**，展示当前行的全量字段信息，字段与列表、表单完全对应。
   - 详情抽屉必须分模块布局（基础信息、关联数据、操作记录等），信息层级清晰，重点信息突出，无字段遗漏。
   - 详情抽屉底部必须包含「编辑」「关闭」按钮，支持从详情直接跳转编辑抽屉，编辑提交后，详情数据实时更新。
   - 详情抽屉必须展示该条数据的关联子数据列表（如订单详情对应的商品列表、设备详情对应的告警记录），数据与主信息联动。
7. **分页功能**
   - 必须包含完整分页控件，支持页码切换、每页条数切换（10/20/50/100 条）、总条数展示、首页 / 末页快速跳转。
   - 分页数据计算准确，筛选 / 新增 / 删除后分页状态实时同步更新，无页码错乱。

#### 3.1.3 可选扩展功能

- 支持表格列宽拖拽、列显隐控制、列排序功能
- 支持行内编辑功能，无需抽屉即可快速修改单字段
- 支持数据冻结（首列 / 表头冻结），适配长表格场景
- 支持表格数据导出 Excel、打印功能
- 支持行数据启用 / 禁用状态切换
- 支持表格数据全屏展示

#### 3.1.4 强制校验标准

✅ 增删改查全流程可完整操作，无断点、无逻辑 BUG

✅ 新增 / 编辑 / 详情 / 高级筛选均使用右侧抽屉实现，仅删除确认使用居中弹窗

✅ 抽屉结构规范，打开关闭动画正常，数据回填 100% 准确

✅ 表单、列表、详情字段 100% 对应，无错配、遗漏

✅ 所有不可逆操作均有二次确认弹窗，无风险操作

✅ 所有操作后数据实时更新，刷新页面数据不丢失

✅ 表单校验逻辑完整，错误提示清晰，提交按钮状态联动正常

✅ 分页、筛选功能准确，无数据错乱

✅ 无控制台报错、无样式错乱、无操作无反馈的情况

### 3.2 图表类页面（全交互可视化）

#### 3.2.1 核心要求

图表必须完整可渲染、全交互可操作，支持类型切换、系列显隐、数据导出、点选交互，浏览器打开即可看到完整图表效果，所有功能可直接操作；**数据明细、图表配置、高级筛选功能必须使用右侧抽屉实现**。

#### 3.2.2 必选功能

1. **基础渲染要求**

   - 必须使用 Chart.js/ECharts 实现，与组件库 CDN 资源兼容，浏览器打开即可自动完整渲染，无控制台报错。
   - 图表必须包含完整元素：X 轴 / Y 轴（含刻度、标签、单位）、图例、数据标签、鼠标悬浮 tooltip、网格线、图表标题 / 副标题。
   - 内置真实合理的模拟数据，不少于 6 组时间 / 维度数据，数据范围统一，无异常值、无意义数据，配色优先复用组件库全局主题色。
   - 图表容器宽高合理，自适应父容器宽度，无溢出、截断、压扁，响应式适配屏幕尺寸。

2. **图表类型切换功能**

   - 必须支持**一键切换图表类型**，核心切换范围：柱状图↔折线图↔面积图↔横向柱状图，堆叠模式↔非堆叠模式。
   - 类型切换后，原数据不丢失、不错乱，坐标轴自适应调整，图例、tooltip 同步更新，切换过程无卡顿、无渲染异常。
   - 切换按钮必须可视化展示，选中状态高亮，清晰标识当前图表类型。

   

3. **数据系列显隐控制**

   - 必须支持点击图例切换对应数据系列的**显示 / 隐藏**，隐藏后图表自动自适应缩放，坐标轴范围同步调整，tooltip 同步过滤隐藏系列。
   - 支持「一键重置」功能，点击恢复所有系列的显示状态，无状态残留。
   - 图例选中 / 未选中状态有明确的视觉区分，交互反馈清晰。

4. **数据点交互功能**

   - 必须支持 hover 数据点 / 柱子 / 扇区，展示完整 tooltip，包含：维度名称、数值、单位、同比 / 环比数据、数据时间。
   - 必须支持点击数据点 / 柱子 / 扇区，打开**右侧抽屉**展示该数据的全量明细信息、关联数据、趋势分析。
   - 支持图表区域缩放、框选数据范围，放大查看细节数据。

5. **导出功能**

   - 必须支持**一键导出**功能，导出格式包含：PNG 高清图片、JPG 图片、PDF 文件。
   - 导出文件命名规范：`[图表名称]_[导出时间].后缀`，导出内容包含图表标题、数据、图例，无截断、无模糊，支持添加水印。
   - 支持「复制图表到剪贴板」功能，复制后可直接粘贴到文档 / PPT 中。

## 四、通用功能强制要求

本章节所有要求，适用于所有原型页面，AI 生成时必须 100% 遵守，违反任意一条均视为无效输出。

### 4.1 交互与反馈规范

1. 所有可操作元素必须有明确的状态样式：正常状态、hover 状态、active 状态、禁用状态，视觉区分清晰。
2. 所有操作执行后，必须有明确的结果反馈：成功 toast、失败 alert、警告提示、加载状态，禁止操作无反馈。
3. 所有加载场景必须展示加载动画：按钮加载、表格加载、页面加载、图表加载，禁止加载无提示。
4. 所有不可逆操作（删除、提交、修改、批量处理），必须有二次确认弹窗，明确提示操作后果，禁止无确认直接执行。
5. 所有右侧抽屉必须支持多种关闭方式：右上角关闭按钮、取消按钮、点击遮罩关闭、ESC 快捷键关闭，形成交互闭环。
6. 所有输入类控件必须支持 Tab 键聚焦、Enter 键确认，基础键盘操作无障碍。
7. 抽屉打开时，禁止背景页面滚动；抽屉关闭后，必须重置表单 / 状态，无数据残留。

### 4.2 数据与存储规范

1. 所有原型必须使用`localStorage`存储模拟数据，刷新页面后数据不丢失，操作记录可追溯。
2. 所有字段命名必须语义化，表单、列表、详情的字段名、字段值 100% 对应，无错配、无类型错误。
3. 模拟数据必须符合业务场景逻辑，数值、时间、状态真实合理，无空数据、无意义数据、逻辑矛盾数据。
4. 数据操作必须原子化，新增 / 编辑 / 删除仅影响目标数据，不改动其他无关数据，无数据污染、丢失。

### 4.3 技术与兼容规范

1. 所有原型必须为**单 HTML 文件**，CSS+JS 全内置，仅可引入以下开源 CDN 资源，禁止引入其他本地 / 外部依赖：
   - 图表库：`https://cdn.jsdelivr.net/npm/chart.js@4.4.8/dist/chart.umd.min.js`
   - 图标库：`https://unpkg.com/lucide@latest/dist/umd/lucide.js`
   - 可选 ECharts：`https://cdn.jsdelivr.net/npm/echarts@5.4.3/dist/echarts.min.js`
2. 必须兼容 Chrome、Edge、Firefox 主流浏览器，无浏览器兼容问题。
3. 必须实现响应式布局，适配桌面端不同屏幕尺寸，无样式溢出、错乱、遮挡。
4. 代码必须符合《UI 组件库 AI 维护规范手册》的命名规范，类名使用专属前缀，注释清晰，无冗余代码，无控制台报错。
5. 所有文本内容必须预留多语言适配能力，使用语义化 key 标识，兼容之前的中英切换功能。

### 4.4 弹出内容使用红线规范

1. ❌ 绝对禁止：新增、编辑、详情、表单、高级筛选等业务内容使用居中弹窗
2. ❌ 绝对禁止：抽屉与弹窗混用，无规范的弹出形式

5. ❌ 绝对禁止：抽屉出现被页面元素遮挡、滚动异常、数据回填错误
6. ❌ 绝对禁止：超过 2 层抽屉叠加嵌套
